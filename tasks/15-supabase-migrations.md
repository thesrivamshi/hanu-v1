# 15 — Migration discipline: from snapshot to versioned migrations

**Priority:** P2 (becomes critical the moment the DB has irreplaceable data)
**Effort:** half a day initial setup, then ongoing
**Depends on:** none (run before tasks 07/08/09/10/11/13/14 ideally — those migrations should be the first numbered files)
**Status:** TODO
**Risk if skipped:** every schema change today rewrites `supabase/schema.sql` as a single snapshot. Re-running it on a populated DB is destructive (some `do $$ ... $$` policy blocks fail re-application; column type changes lose data). Once memories, conversations, and decisions accumulate, an incorrect schema PR costs the user real loss.

---

## Context

Today there is exactly one file: `supabase/schema.sql`. It's the initial snapshot. Every change has been applied by editing this file and re-running it manually. There is no version table, no migration history, no `down` migrations.

Supabase provides `supabase migration new` and the `supabase` CLI to manage this. The same project also tracks migration state in `supabase_migrations.schema_migrations`. Adopting it now (while the DB is essentially empty) is free; adopting it after data accumulates is painful.

---

## Acceptance criteria

- The Supabase CLI is installed (`supabase --version` returns a value).
- A `supabase/` directory follows the CLI's expected layout (`supabase/migrations/` for migrations, `supabase/config.toml` for project config).
- The current `schema.sql` is captured as the initial migration: `supabase/migrations/00000000000000_initial_schema.sql`.
- Future schema changes go in new files generated by `supabase migration new <name>`.
- `supabase db push` applies pending migrations to the live project; `supabase db diff` detects drift.
- README updated to document the workflow.

## Implementation steps

### Step 1 — Install the Supabase CLI

```bash
# macOS (dev machine):
brew install supabase/tap/supabase

# Linux (droplet):
curl -fsSL https://supabase.com/install.sh | sh
# or download the binary from https://github.com/supabase/cli/releases
```

Verify:

```bash
supabase --version
```

### Step 2 — Initialize the project structure

In the repo root:

```bash
cd /Users/srivamshi/MyDrafts/Hanu-v1
supabase init
```

This creates:

```
supabase/
├── config.toml         # project-local settings (gitignored sections handled inside)
├── migrations/         # empty (we'll add the seed below)
├── functions/          # for edge functions (unused for v1)
└── seed.sql            # optional starter data (currently empty)
```

The existing `supabase/schema.sql` is preserved; we will use it as the source for the first migration.

### Step 3 — Link the local project to the remote

```bash
supabase link --project-ref lcayzfqmemitlbjugbsq
```

You'll be prompted for the DB password. Use the one stored in `.env.local` as `SUPABASE_DB_PASSWORD` (or fetch from Supabase dashboard → Project Settings → Database).

### Step 4 — Capture the current state as the initial migration

Two options:

**Option A — Use the existing snapshot:**

```bash
# Rename the existing schema.sql so the CLI's migrate workflow doesn't clash:
mkdir -p supabase/migrations
mv supabase/schema.sql supabase/migrations/00000000000000_initial_schema.sql
```

Edit the file header to add a migration comment. Mark this migration as already-applied in the remote DB:

```bash
supabase migration repair --status applied 00000000000000
```

This tells the CLI's migration tracking table that the initial migration has already been applied (it was, manually) so future pushes don't try to re-run it.

**Option B — Generate the initial migration by introspection:**

```bash
supabase db pull --schema public > supabase/migrations/00000000000000_initial_schema.sql
supabase migration repair --status applied 00000000000000
```

This produces a CLI-generated equivalent. Slightly more accurate (matches actual remote state) but less readable than the hand-authored snapshot. **Option A** is recommended because the snapshot includes carefully-written WHY comments that introspection drops.

### Step 5 — Workflow for new changes

Every task that touches the schema (07, 08, 09, 10, 11, 12, 13, 14, 22, etc.) should now follow this pattern:

```bash
# Generate a new migration file:
supabase migration new privacy_levels_remap
# Creates: supabase/migrations/20260513154500_privacy_levels_remap.sql

# Edit the file: put the SQL from the task in there. Include both UP changes
# and (in a -- DOWN comment block) the rollback.

# Apply locally to a dev DB:
supabase db reset    # resets local Postgres and runs all migrations from scratch

# When happy, push to remote:
supabase db push     # applies only the new migration
```

For the migrations that include destructive transforms (privacy remap, enum migrations), wrap them in an explicit `BEGIN ... COMMIT` so a mid-file failure leaves the DB untouched.

### Step 6 — Detect drift

```bash
supabase db diff
```

Returns SQL needed to bring the remote schema in line with the local migrations. Run it as a pre-commit check (CI later) so that someone making changes through the Supabase dashboard doesn't go undetected.

### Step 7 — Update `.gitignore`

```
supabase/.temp/
supabase/.branches/
supabase/.env
```

Ensure `supabase/config.toml`, `supabase/migrations/`, and `supabase/seed.sql` ARE committed.

### Step 8 — Document the workflow in CLAUDE.md

Append a section under "Editing conventions specific to this repo":

```markdown
- Schema changes go through Supabase CLI migrations. Never edit
  `supabase/migrations/*.sql` after a migration has been applied. To change a
  prior migration, write a NEW migration that alters the affected objects.
  Use `supabase migration new <name>` to generate.
- Apply pending migrations with `supabase db push`. Detect drift with
  `supabase db diff` before committing schema-touching PRs.
```

### Step 9 — Seed data

`supabase/seed.sql` is for "data the DB needs even when empty" (e.g., the user's profile row, default `settings`). Move the bootstrap inserts from manual `psql` invocations into this file:

```sql
-- seed.sql — ran after migrations on `supabase db reset` for local dev
insert into public.profiles (id, display_name, first_name, avatar_letter)
values ('d804b9ed-5eaa-497c-8390-86ba02007a33', 'Vamshi', 'Vamshi', 'V')
on conflict (id) do nothing;

insert into public.settings (user_id) values
  ('d804b9ed-5eaa-497c-8390-86ba02007a33')
on conflict (user_id) do nothing;

insert into public.spaces (user_id, name, kind, description)
values
  ('d804b9ed-5eaa-497c-8390-86ba02007a33', 'My private space', 'private', NULL)
on conflict do nothing;
```

Note: `seed.sql` only runs on local resets, not against the remote. Remote one-time inserts (like the user's `auth.users` row) go through the Supabase dashboard or are documented in a separate `supabase/setup-notes.md`.

## Verification

```bash
# Local DB reset and check that all migrations apply cleanly:
supabase db reset

# Confirm remote drift is zero:
supabase db diff
# Expect: empty output

# List applied migrations:
supabase migration list
```

End-to-end: write a tiny test migration:

```bash
supabase migration new test_drop_me
echo "create table public._test_drop_me (id int);" >> supabase/migrations/*_test_drop_me.sql
echo "drop table public._test_drop_me;" >> supabase/migrations/*_test_drop_me.sql
supabase db push
# Then delete the file and `supabase db push` again to confirm idempotency.
```

## Rollback

If the CLI initialization goes wrong:
- Restore `supabase/schema.sql` from git (it was just renamed, not deleted).
- Delete `supabase/migrations/`, `supabase/config.toml`.
- Continue manual workflow as before.

Don't roll back the migration history in production — it'd just mean re-applying any migrations the CLI thinks weren't applied yet.

## Notes

- The Supabase CLI is also the entry point to local dev: `supabase start` runs a local Postgres + Auth + Storage stack on the machine. This lets you test migrations without hitting production. Worth setting up after task 15 is done.
- After this task, every later schema-touching task should be re-expressed as a numbered migration file rather than a `schema.sql` edit. Update the per-task instructions accordingly when you get to them.
- Migration filename convention: `YYYYMMDDHHMMSS_descriptive_name.sql`. The CLI handles the timestamp.
- For destructive transformations (column type changes), use the temporary-column pattern: add `_v2` column, backfill, drop old, rename. Documented in task 07.
